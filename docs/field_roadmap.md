This document defines the full field universe for solar permit and SolarAPP approval document intelligence.

**See also:**
- `ROADMAP.md` - phased implementation plan

---

## Document Classes

This project supports two document classes:

1. **Permit Packets** - submitted by contractors to AHJs
   - Semi-structured application forms
   - Electrical diagrams, site plans, equipment specs
   - Varies by jurisdiction

2. **SolarAPP Approval Documents** - generated by SolarAPP+ systems
   - Structured approval/inspection checklists
   - Normalized equipment specifications
   - Standardized electrical service details

## Schema Evolution Plan

### Phase 5: AC/DC System Size Split (BREAKING)

**Before:**
```python
{
  "system_size_kw": 12.48  # Ambiguous: AC or DC?
}
```

**After:**
```python
{
  "system_size_kw": 12.48,     # Deprecated, aliased to system_size_dc_kw
  "system_size_ac_kw": 11.4,   # Inverter capacity
  "system_size_dc_kw": 12.48   # Array capacity
}
```

**Migration steps:**
1. Add new fields alongside old field (Phase 5)
2. Update all truth fixtures
3. Update extraction logic to populate both
4. Update validation logic to use new fields
5. Mark `system_size_kw` as deprecated
6. Remove `system_size_kw` in Phase 6

---

### Phase 4: Equipment Manufacturer Splits (ADDITIVE)

**Before:**
```python
{
  "inverter_model": "SolarEdge SE7600H-US",
  "battery_model": "Tesla Powerwall 2"
}
```

**After:**
```python
{
  "inverter_model": "SolarEdge SE7600H-US",        # Keep for backward compat
  "inverter_manufacturer": "SolarEdge",            # New
  "inverter_architecture": "string",               # New
  "battery_model": "Tesla Powerwall 2",            # Keep
  "battery_manufacturer": "Tesla"                  # New
}
```

**Why additive:**
- Existing fields remain
- New fields are optional
- No breaking changes to current fixtures

---

### Phase 3: Project Type Enum (ADDITIVE)

**New field:**
```python
{
  "project_type": "residential"  # enum: residential, commercial, ground_mount, carport, agricultural
}
```

**Why additive:**
- New field, no conflicts
- Can be null if not extracted
- Enables type-based validation immediately
